# 11 — Non-OData Additions (FileMaker-Specific Extensions)

This document summarizes all FileMaker-specific extensions that go beyond the OData 4.0 standard. These are the features that make the FileMaker OData API unique and that wrapper libraries must handle specially.

## Custom `Prefer` header values

FileMaker defines FileMaker-specific values for the standard `Prefer` header. These are not part of the OData 4.0 specification.

| Prefer value | Effect |
|--------------|--------|
| `fmodata.basic-timestamp` | Respond with FileMaker timestamp format instead of Zulu time format |
| `fmodata.gmtoffset` | Respond with timestamp difference from Greenwich Mean Time |
| `fmodata.entity-ids` | Respond with table and field IDs instead of names |
| `fmodata.include-specialcolumns` | Respond with `ROWID` and `ROWMODID` system fields |

### Standard `Prefer` values (also supported)

| Prefer value | Source |
|--------------|--------|
| `odata.continue-on-error` | OData 4.01 |
| `odata.maxpagesize` | OData 4.01 |
| `return=representation` | OData 4.01 |
| `return=minimal` | OData 4.01 |

## Metadata annotations

FileMaker adds non-standard annotations to the `$metadata` CSDL document. These are not part of the OData standard. See [docs/05-metadata.md](05-metadata.md) for the full list.

Key annotations:
- `AutoGenerated`, `Index`, `VersionID`, `Global`, `Calculation`, `Summary`, `Computed` (boolean field properties)
- `MaxRepetitions`, `ExternalSecurePath` (value annotations)
- `TableID` (FMTID), `FieldID` (FMFID), `ScriptID` (FMSID) (immutable IDs)
- `FMComment`, `AIAnnotation` (descriptive metadata)
- `BestRowID` (ROWID), `RowVersion` (ROWMODID) (system fields)
- `Org.OData.Core.V1.ProductVersion`, `ServerVersion` (version detection)

## Container field annotations (JSON body)

When uploading container data via base64 JSON, FileMaker uses custom annotations:

| Annotation | Description |
|------------|-------------|
| `@com.filemaker.odata.Filename` | Filename to store in the container |
| `@com.filemaker.odata.ContentType` | MIME type of the data |

## System fields

| Field | Description | Equivalent FileMaker function |
|-------|-------------|-------------------------------|
| `ROWID` | Record ID | `Get(RecordID)` |
| `ROWMODID` | Record modification count | `Get(RecordModificationCount)` |

These are not standard OData properties — they are FileMaker-specific system fields accessed via `$select`.

## Immutable IDs

FileMaker provides immutable IDs that survive renames:

| ID | Abbreviation | Used in URL as |
|----|-------------|----------------|
| Table ID | FMTID | `FMTID:<id>` (replaces table name) |
| Field ID | FMFID | Used in `$select`, `$orderby`, `$filter` (replaces field name) |
| Script ID | FMSID | `Script.FMSID:<id>` (replaces script name) |

These are FileMaker-specific and not part of the OData standard.

## FMID authentication scheme

FileMaker Cloud uses the `FMID` authentication scheme:

```
Authorization: FMID <Claris_ID_Token>
```

This is a FileMaker-specific auth scheme, not a standard HTTP auth scheme. See [docs/04-authentication.md](04-authentication.md).

## Script system table

Scripts are exposed as actions on a system table named `Script`. This is a FileMaker-specific convention — OData actions are not typically bound to a "system table" concept.

## System tables for DDL

FileMaker uses system tables (`FileMaker_Tables`, `FileMaker_Indexes`) as the target for DDL operations. This is a FileMaker-specific approach — standard OData does not define DDL via system tables.

## Webhook system actions

Webhook management is exposed via FileMaker-specific actions (`Webhook.Add`, `Webhook.Remove`, `Webhook.Get`, `Webhook.GetAll`, `Webhook.Invoke`). These are not OData standard actions.

## Value list enum types

FileMaker value lists appear as `EnumType` in metadata with a FileMaker-specific naming convention:

```
com.filemaker.odata.<database-name>.<value-list-name>
```

Fields using a value list contain the `EnumType` property referencing this name.

## Implications for wrapper libraries

These non-OData additions mean that a generic OData client library **cannot** fully interact with FileMaker Server. A FileMaker-aware wrapper must:

1. Parse FileMaker-specific metadata annotations.
2. Support the `Script.<name>` / `Script.FMSID:<id>` action pattern.
3. Handle container field binary and base64 upload/download.
4. Support the `FMID` auth scheme (for FileMaker Cloud).
5. Send FileMaker-specific `Prefer` header values when needed.
6. Expose system fields (`ROWID`, `ROWMODID`) via `$select`.
7. Support immutable IDs (FMTID, FMFID, FMSID) for stable integrations.
8. Handle webhook management operations.
9. Support DDL operations via system tables (with appropriate safety gates).
10. Handle the `/$count` suffix not being supported (use inline `?$count=true`).
