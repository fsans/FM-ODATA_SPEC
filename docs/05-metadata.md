# 05 — Metadata, Annotations, and System Tables

The `$metadata` endpoint returns the complete schema of a database in CSDL (Common Schema Definition Language) / EDMX format (XML). FileMaker adds non-standard annotations to provide FileMaker-specific information.

## Get metadata

### Request

```
GET /fmi/odata/v4/<database>/$metadata
Authorization: <auth>
```

### Response

XML document in EDMX format containing:
- Entity types (tables) with properties (fields)
- Entity sets (table occurrences)
- Actions (scripts)
- Enum types (value lists)
- FileMaker-specific annotations

## FileMaker metadata annotations

These annotations are not part of the OData standard. They provide FileMaker-specific field and table metadata.

### Boolean field annotations

Present with value `true` if the condition holds; absent (or `false`) otherwise.

| Annotation | Description |
|------------|-------------|
| `AutoGenerated` | Field value is automatically generated by FileMaker Pro |
| `Index` | Field values are indexed |
| `VersionID` | Field is a timestamp field; new timestamp generated on record modification |
| `Global` | Field contains a global value |
| `Calculation` | Field is a calculation type |
| `Summary` | Field is a summary type |
| `Computed` | Field is a calculation or derives value from external sources via auto-enter options |

### Value annotations

| Annotation | Type | Description |
|------------|------|-------------|
| `MaxRepetitions` | Integer | Maximum repetitions for a repeating field. Absent if not a repeating field. |
| `ExternalSecurePath` | String | Relative path for secure external storage of a container field |
| `BestRowID` | — | Always `ROWID`; system field included via `$select=ROWID` |
| `RowVersion` | — | Always `ROWMODID`; system field included via `$select=ROWMODID` |
| `TableID` | String | Immutable FileMaker Table ID (FMTID) |
| `FieldID` | String | Immutable FileMaker Field ID (FMFID) |
| `ScriptID` | String | Immutable FileMaker Script ID (FMSID) |
| `FMComment` | String | Table or field comment |
| `AIAnnotation` | String | Additional field info describing its purpose for AI models (Claris 2026+) |

### Product version annotation

The metadata includes the FileMaker product version number via one or more XML annotations. Downstream libraries should parse these using a multi-strategy approach, as the exact format varies across FileMaker Server versions and XML serializers:

**Strategy 1a — Canonical ProductVersion annotation (all versions):**
```xml
<Annotation Term="Org.OData.Core.V1.ProductVersion" String="21.1.2.500"/>
```

**Strategy 1b — Reversed attribute order:**
Some XML serializers emit `String` before `Term`:
```xml
<Annotation String="21.1.2.500" Term="Org.OData.Core.V1.ProductVersion"/>
```

**Strategy 1c — ServerVersion annotation (Claris 2026+):**
FM 26+ adds a separate annotation with a descriptive prefix:
```xml
<Annotation Term="ServerVersion" String="OData Engine 26.0.1"/>
```

**Strategy 1d — ServerVersion with reversed attribute order:**
```xml
<Annotation String="OData Engine 26.0.1" Term="ServerVersion"/>
```

**Strategy 2 — Generic fallback:**
If none of the above match, look for any annotation whose term name contains "Version" and whose String value contains a version number with major >= 17 (the threshold avoids false positives from the OData spec version "4.0" in the `edmx:Edmx` root element).

**Version string format:**
The version string may include a build number suffix (e.g. `"21.1.2.500"`). Parse the first three numeric segments as `major.minor.patch` and ignore the rest.

**Implementation reference:**
The `@fms-odata/spec-ts` package provides `parseServerVersion(metadataXml)` which implements all 4 strategies plus the generic fallback, returning an `FMServerVersion` object `{ major, minor, patch, raw }`. Downstream libraries should use this function rather than reimplementing the parsing logic.

## Value lists in metadata

Value lists appear as `EnumType` in the metadata:
- The `Kind` property has the value `EnumType`.
- Fields using a value list contain the `EnumType` property with the value `com.filemaker.odata.<database-name>.<value-list-name>`.

## Primary key handling

OData requires each table to define a primary key. FileMaker uses fields that are:
- Not empty, AND
- Require a unique value

If no such field is defined, the **`ROWID` system field** is used as the primary key. `ROWID` contains the same value as `Get(RecordID)` for the record.

## System fields

| Field | Description | How to access |
|-------|-------------|---------------|
| `ROWID` | Record ID (same as `Get(RecordID)`) | `$select=ROWID` |
| `ROWMODID` | Record modification count (same as `Get(RecordModificationCount)`) | `$select=ROWMODID` |

These fields are not returned by default — they must be explicitly requested via `$select`.

## System tables

FileMaker exposes system tables via OData for both querying and schema modification.

| System table | Purpose |
|--------------|---------|
| `FileMaker_Tables` | List, create, modify, and delete tables |
| `FileMaker_Fields` | List fields in tables |
| `FileMaker_Indexes` | Create and delete field indexes |
| `FileMaker_Scripts` | List scripts (via metadata, not directly queryable as a table) |

### Querying system tables

```
GET /fmi/odata/v4/<database>/FileMaker_Tables
GET /fmi/odata/v4/<database>/FileMaker_Fields
```

### Using system tables for schema modification

System tables also serve as the target for DDL operations. See [docs/10-schema-modification.md](10-schema-modification.md).

## Immutable IDs

FileMaker provides immutable IDs that survive renames. These are critical for stable integrations:

| ID type | Abbreviation | Found in | Used for |
|---------|-------------|----------|----------|
| Table ID | FMTID | Metadata `TableID` annotation | Replace table name in URL paths |
| Field ID | FMFID | Metadata `FieldID` annotation | Replace field name in `$select`, `$orderby`, `$filter` |
| Script ID | FMSID | Metadata `ScriptID` annotation | Replace script name in `Script.FMSID:<id>` |

### Using IDs instead of names

```
# By name
GET /fmi/odata/v4/MyDB/Contacts
POST /fmi/odata/v4/MyDB/Script.HelloScript

# By ID (Claris 2026+)
GET /fmi/odata/v4/MyDB/FMTID:123
POST /fmi/odata/v4/MyDB/Script.FMSID:456
```

Using IDs is recommended for production integrations because they survive renames of tables, fields, and scripts.

## Wrapper library guidance

Downstream libraries should:

1. Parse `$metadata` to extract entity types, entity sets, actions, and annotations.
2. Cache metadata per session (it's expensive to fetch and rarely changes).
3. Provide a method to refresh metadata on demand.
4. Extract and expose FileMaker annotations (FMTID, FMFID, FMSID, comments, AI annotations).
5. Use the product version annotation to detect the FileMaker Server version and gate features.
6. Handle both `Edm.Stream` and `Edm.Binary` types for container fields (different versions emit different types).
7. Auto-resolve non-ASCII field names to FMFID to avoid encoding issues in `$filter` (Claris 2026+).
